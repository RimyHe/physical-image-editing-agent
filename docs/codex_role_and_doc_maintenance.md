# Codex 角色与文档维护约定

## 1. 适用范围

本约定适用于 Codex 在 `physical_image_editing_agent` 项目目录下工作时的默认行为。

## 2. Codex 的工作角色

Codex 在本项目中不只是代码实现助手，也承担持续维护项目文档的职责。除了完成具体开发任务外，还应定期把稳定结论、失败经验、设计变化和技术取舍整理进 `docs/`，避免信息只停留在对话或临时输出里。

## 3. 文档维护原则

在该目录下工作时，Codex 应经常维护 `docs` 中的各类文档，并及时更新、整理，而不是等到阶段结束后再一次性补写。

维护要求：

1. 遇到重要错误、评测偏差、实现弯路或错误假设时，应及时记录原因、影响和修正方案。
2. 当计划、阶段重点或实现路线发生变化时，应同步更新开发计划，而不是让计划文档长期滞后。
3. 当项目中的功能角色、模块边界、运行方式或 Codex 的职责发生变化时，应同步更新角色说明。
4. 当完成新的论文、框架、API 或 benchmark 调研时，应及时整理为技术调研文档或补充到现有文档。
5. 优先更新已有文档；只有当现有文档没有合适位置时，才新增文档，避免主题重复和内容分散。

## 4. 文档分工

### 4.1 进展汇报

侧重记录：

- 遇到的错误
- 走过的弯路
- 失败案例
- 不可靠的评测结果
- 问题是如何定位和解决的

推荐文档：

- `physical_dataset_progress_report.md`

### 4.2 开发计划

侧重记录：

- 当前阶段目标
- 下一步任务拆解
- 计划中的模块演进
- 尚未实现但已决定的技术路线

推荐文档：

- `development_plan.md`

### 4.3 Codex 角色

侧重记录：

- Codex 在项目中的职责
- 功能角色边界
- 运行和协作约定
- 文档维护责任

推荐文档：

- `codex_role_and_doc_maintenance.md`
- `task_brief.md`

### 4.4 技术调研

侧重记录：

- API 能力和限制
- Agent 架构调研
- 工具路由策略
- benchmark / 评测协议
- 外部论文和参考项目的可借鉴结论

推荐文档：

- `closed_model_api_pipeline_research.md`
- `tool_router_physics_tool_research.md`

## 5. 何时必须更新文档

出现以下情况时，Codex 应优先考虑同步更新 `docs/`：

1. 新完成一项可复用的实现或脚本。
2. 发现当前评测流程、数据处理或实验记录中存在系统性问题。
3. 跑出一批新的 benchmark 结果，并形成稳定结论。
4. 关键设计假设被推翻，或路线发生明显调整。
5. 新增了以后会反复使用的工具、页面、脚本或工作流。

## 6. 目标

目标不是堆积文档数量，而是让 `docs/` 持续反映项目的真实状态，使后续开发能快速回答以下问题：

- 当前项目做到哪一步了？
- 之前踩过哪些坑？
- 现在的设计为什么是这样？
- 下一步该做什么？
- 某项评测结果是否可靠？
